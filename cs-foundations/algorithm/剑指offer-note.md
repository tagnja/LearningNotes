<h2><a name="intv_proc_t" href="#intv_proc_c">1. 面试流程</a></h2>
<br>

### 1.1 面试官谈面试

- 初级程序员偏向考察算法和数据结构，看应聘者的基本功；对于高级程序员，我会多关注专业技能和项目经验。
- 应聘者要事先做好准备，对公司近况、项目情况有所了解，对应聘的工作很有热情。还要准备好合适的问题问面试官。
- 面试过程中，首先需要放松，其次不要基于编写代码，写完代码不要马上提交，自己检查和测试一下。

### 1.2 面试的3种形式

电话面试

共享桌面远程面试

- 想清楚再开始编码
- 良好的代码命名和缩进对齐习惯。
- 能够进行单元测试。

现场面试，应做好以下准备

- 规划好路线和估算出行时间。
- 准备好得体的衣服。
- 注意面试邀请函里的面试流程。
- 准备几个问题。应聘者提问的问题。

### 1.3 面试的3个环节

#### 行为面试环节

**自我介绍**

应聘者的项目经验

- Situation：简短的项目背景。项目的规模，开发软件的功能，目标用户等。
-  Task：自己完成的任务。参与xxx，负责xxx。
- Action：为完成任务做了哪些工作，怎么做的。详细介绍。基于什么平台，应用了什么技术等。
- Result：自己的贡献。最好能用数字加以说明。完成了多少功能；优化，性能提高了多少百分比；维护，修复了多少个 Bug。

针对项目经验最常问的问题：

- 你在该项目中碰到的最大问题是什么，你是怎么解决的？
- 从这个项目中你学到了什么?
- 什么时候会和其他团队成员（包括开发、测试、设计、项目经理等）有什么样的冲突，你们是怎么解决冲突的？

**应聘者掌握的技能**

- 了解。某项技术上过课或者看过书，没有做过实际的项目。不建议写在简历中，除非对应聘的岗位需要。
- 熟悉。简历中大部分应该是熟悉。在实际项目中使用某项技术很长时间，通过查阅文档可以独立解决大部分问题。毕业设计用到的技术，工作项目开发中用到的技术，可以用熟悉。
- 精通。能够轻松回答这个领域的绝大多数问题。不要轻易在简历上写精通。

**回答为什么跳槽**

避免以下4个原因

- 老板太苛刻。
- 同事太难相处。
- 加班太频繁。
- 工资太低。

正确回答的参考：

- 对现在的工作有些厌倦，没有激情。
- 对某些方面没有太多兴趣。
- 主要工作是维护和修复 Bug，软件开发和设计能力得不到提升。

#### 技术面试环节

5个方面的素质：

- 扎实的基础知识。包括编程语言，数据结构和算法等。
- 高质量的代码。能写出正确的、完整的、鲁棒的高质量代码。
- 清晰的思路。能思路清晰地分析、解决问题。
- 优化效率的能力。能从时间、空间复杂度两方面优化算法效率。
- 优秀的综合能力。具备优秀地沟通能力、学习能力、发散思维能力等。

#### 应聘者提问

不适合问的问题

- 不要问和自己的职位没有关系的问题。如公司未来五年的发展战略是什么。
- 不要问薪水。工资一般是通过面试后跟 HR 谈。
- 不要立即打听面试结果。

推荐问的问题

- 与应聘的职位或项目相关的问题。如，主要做什么工作。需要学习相关的知识吗。
- 了解公司相关信息再提问。如公司成立时间，主要的业务，职位要求等。
- 留心面试官说过的话。

<h2><a name="intv_basic_knowle_t" href="#intv_basic_knowle_c">2. 面试需要的基础知识</a></h2>
<br>

### 2.1 面试官谈基础知识

- C++ 的基础知识，如面向对象、构造函数、析构函数、动态绑定等，能够反映应聘者是否善于把握本质，有没有耐心深入一个问题。另外还有常用的设计模式、UML图等，体现应聘者是否有软件工程的经验。
- 对于基础知识的考察，我特别重视 C++ 中对内存的使用管理。
- 基础知识反映了一个人的基本能力和基础素质。我一般考察：（1）数据结构和算法；（2）编程能力；（3）部分数学知识，如概率；（4）问题分析和推理能力。
- 我比较重视四块基础知识：（1）编程基本功，如字符串处理问题；（2）并发控制；（3）算法、复杂度；（4）语言的基本概念。
- 我会考察编程基础、计算机系统基础知识、算法及设计能力。这些时软件工程师最基本的要求。
- （1）对 OS 的理解度。对于 OS 理解比较深入的人对于偏底层的工作上手一般比较快。（2）对于一门编程语言的掌握程度。对某种语言比较深入了解，对新的编程语言上手也比较快。（3）常用的算法和数据结构。不了解这些只能写写 Hello World。

综合来说

- 数学能力。
- 算法和编程能力。
- 编程语言语法特性的深入理解。
- 计算机基础的深入理解。
- 软件工程能力。
- 其他综合能力。

### 2.2 编程语言

面试中会至少设计一种编程语言。

面试过程中，考察编程语言：要么直接问语言的语法，要么让应聘者用一种编程语言写代码解决一个问题。

流行的编程语言有很多，不同公司开发用的语言也不尽相同。常见的语言和其应用场景如：

- 底层开发：C。
- Linux 下开发：C++。
- 基于 Windows：C#。
- 跨平台开发：Java。
- 基于苹果系统开发：Objective C。
- 短小精致的小应用软件：脚本语言 Perl、Python。

### 2.3 数据结构

数据结构一直是技术面试的重点，面试大多数围绕数组、字符串、链表、树、栈及队列这几种常见的数据结构。每一个都需要熟练掌握。

数组和字符串是两种最基本的数据结构，它们是连续内存。链表和树是面试种出现频率最高的数据结构。由于操作链表和树需要大量的指针，一定要留意代码的鲁棒性。栈和队列可以帮助我们高效的解决很多问题。

数组

- 由于数组种的内存是连续的，可以根据下标 O(1) 时间读写任何元素，可以使用数组来是实现简单的哈希表。

链表

- 链表的结构很简单，代码量比较适合面试。
- 链表是一种动态的数据结构，需要比较好的编程功底才能写出完整的操作链表的代码。
- 链表比较灵活，可以用链表来设计具有挑战性的面试题。

树

- 树的操作会涉及大量的指针，面试中往往利用树来考察在复杂指针操作情况下写代码的能力。面试中大部分是二叉树。
- 二叉树有3种遍历方式：前序、中序和后序。每一种哦遍历都有递归和循环两种不同的实现方式。另一种，按层遍历比较少用。
- 二叉树有很多特例。如二叉搜索树，堆，和红黑树。

### 2.4 算法和数据操作

- 很多算法都可以用递归和循环两种不同的实现方式。通常基于递归的实现方式代码会比较简洁，但性能不如基于循环的实现方式。
- 通常排序和查找是面试时的考察算法的重点。重点掌握二分查找、归并排序和快速排序，做到能随时正确、完整地写出它们地代码。
- 如果要求在二维数组（表现为迷宫或者棋盘等）上搜索路径，我们可以尝试使用回溯法。回溯法很适合用递归的代码实现，我们也可以用栈来模拟递归过程。
- 如果要求某个问题的最优解，并且该问题可以分为多个子问题，那么我们可以尝试用动态规划。
- 如果在分解子问题的时候存在某个特殊的选择，采用这个选择一定能得到最优解，那么，可以尝试使用贪婪算法。
- 位运算可以看成一类特殊的算法。位运算有与、或、异或、左移和右移5种位运算。

循环和递归

- 递归的代码比较简洁，但性能没有循环好。如果面试官没有特殊的要求，应聘者可以优先采用递归的方式编程。
- 递归的缺点表现为：每一次函数调用，都需要在内存栈中分配空间以保存参数、返回地址和临时变量，而且往栈里压入数据和弹出数据都需要时间。递归还有可能导致调用栈溢出。

查找和排序

- 查找相对比较简单，不外乎顺序查找、二分查找、哈希表查找和二叉树查找。
- 排序比查找要复杂一些。面试总经常要求比较插入排序、冒泡排序、归并排序、快速排序等不同算法的优劣。
- 面试时，一定要对各种排序算法的特点非常熟悉，能够从额外空间消耗、平均时间复杂度和最差时间复杂度等方面比较它们的优缺点。
- 面试经常要求写出快速排序的代码。

<h2><a name="high_qual_code_t" href="#high_qual_code_c">3. 高质量代码</a></h2>
<br>

### 3.1 面试官谈高质量的代码

- 一般考察应聘者对代码的容错处理能力。不能容忍只针对假想的正常值进行处理，不考虑异常状况，也不考虑资源回收等问题。
- 如果是粗心犯错，则可以原谅，因为毕竟面试的时候会紧张；不能容忍的是，该掌握的知识点却没有掌握，而且提醒了还不知道。如 double d1,d2; if(d1 == d2)
- 最不能容忍功能错误，忽略边界情况。
- 如果一个程序员连变量、函数命名都毫无章法，解决一个具体的问题都找不到一个最合适的数据结构，那么这回让面试官对他的印象大打折扣。这只能说明他程序写得太少，不够熟悉。
- 我会从程序的正确性和鲁棒性两个方面检验代码的质量。会关注对输入参数的检查、处理错误和异常的方式、命名方式等。

### 3.2 代码的规范性

- 清晰的书写。手写代码时，减慢速度，写清楚时很有必要的。
- 清晰的布局。合理的缩进，括号对齐。
- 合理的命名。用完整的英文单词组合命名变量和函数。

### 3.3 代码的完整性

- 完成了基本的功能。
- 输入边界值能够得到正确的输出。
- 对不合规范的非法输入做出合理的错误处理。

保证代码完整性的测试

- 功能测试。
- 边界测试。输入的边界值是否得到正常输出。代码中的循环的边界是否正确。递归终止的边界值是否正确。字符串转换为数字和数字算术运算是否溢出数值边界。
- 负面测试。考虑错误输入。

错误处理的方法

- 函数用返回值告知调用者是否出错。很多 API 返回值为0表示成功，不为0表示错误。
- 设置一个全局变量。
- 抛出异常。有些编程语言不支持抛出异常。

### 3.4 代码的鲁棒性

鲁棒性（Robust）是指程序能够判断输入是否合乎规范要求，并对不符合要求的输入予以合理的处理。它属于完整性中的负面测试。

总结

高质量代码主要考虑规范性和完整性（功能、边界、负面）。

<h2><a name="solution_t" href="#solution_c">4. 解决面试题的思路</a></h2>
<br>

### 4.1 面试官谈面试思路

- 我个人比较倾向于让应聘者在写代码之前解释他的思路。
- 对于比较复杂的算法和设计，一般来说最好在开始写代码之前讲清楚思路和设计。

### 4.2 画图让抽象问题形象化

画图是帮助自己分析、推理的常用手段。很多面试题很抽象，不容易找到解决方法。这时，不妨画一些与题目相关的额图形，借以辅助自己观察和思考。图形能使抽象的问题具体化、形象化。

二叉树、二维数组、链表等问题可以采用画图的方法来分析。

在面试中，可以画图来解释自己的思路。

### 4.3 举例让抽象问题具体化

我们也可以借助举例模拟的方法来思考分析复杂的问题。当一眼看不出问题中隐藏的规律时，我们可以试着用一两个具体的例子模拟操作的过程，说不定能通过具体的例子找到抽象的规律。

### 4.4 分解让复杂问题简单化

在面试中，当我们遇到复杂的大问题的时候，如果能够先把大问题分解成若干个简单的小问题，然后再组个解决这些小问题，则可能会容易很多。我们可以按照解决问题的步骤来分解复杂问题，每一步解决一个小问题。

<h2><a name="opt_t" href="#opt_c">5. 优化时间和空间效率</a></h2>
<br>

### 5.1 面试官谈效率

- 通常针对有较长工作经验的应聘者，会问一些关于时间和空间效率的问题，这能够体现一个应聘者较好的编程素质和能力。
- 通过会限定时间和空间复杂度，让应聘者给出解决方案。
- 是空间换时间还是时间换空间，这样看具体的题目。

### 5.2 时间效率

- 每个人都希望软件的响应时间尽量短一些，所以软件公司都很重视软件的时间性能，都会在发布软件之前进行时间效率优化。
- 面试中除了考察应聘者的编程能力，还会关注应聘者有没有不断优化效率、追求完美的态度和能力。
- 代码的时间效率还能体现应聘者对数据结构和算法的掌握程度。同样是查找，如果顺序查找需要 O(n) 的时间；有序数组的二分查找需要 O(logn) 的时间；构造哈希表，查找只需要 O(1) 的时间。

### 5.3 时间效率与空间效率的平衡

- 如果我们分配少量的辅助空间来保存计算的中间结果以提高时间效率，则通常是可以被接受的。
- 以空间换时间的策略并不一定都是可行的，需要具体问题具体分析。
- 很多时候时间和空间是需要有所取舍的，到底如何选择，我们可以和面试官进行讨论和沟通。

<h2><a name="abilities_t" href="#abilities_c">6. 面试中的各项能力</a></h2>
<br>

### 6.1 面试官谈能力

- 应聘者能够礼貌平和、不卑不亢地和面试官交流，逻辑清晰、详略得当地介绍自己以及项目经历，谈论题目时能够发现问题的细节并向面试官进行询问，这些都是比较好的沟通表现。对自己做的项目能够了解得很深入、对面试题能够快速寻找解决方法时判断应聘者学习能力得一个方法。沟通和学习这两个能力都很重要。
- 有时候会问一些应聘者不是很熟悉的领域，看应聘者在遇到难题时的反应。
- 如果应聘者沟通能力不行、难以合作，那么我们也不会录取。
- 沟通、学习能力就是看面试者能否清晰、有条理地表达自己，是否会在自己所得到的信息不足的情况下主动发问，能够在得到一些暗示之后迅速做出反应纠正错误。

### 6.2 沟通能力和学习能力

沟通能力

- 随着软件、系统功能越来越复杂，开发团队的规模也随之扩张，开发人员、测试人员和项目经理之间的沟通交流也变得越来越重要。
- 很多公司会考察应聘者的沟通能力，要求应聘者在介绍项目经验和介绍解题思路的时候，逻辑清晰，语言得当，表述的时候突出重点、观点明确。
- 不能把好的沟通能力理解成夸夸其谈。**在面试中，知之为知之，不知为不知，对于不清楚的知识点，要勇于承认，千万不要不懂装懂。**

学习能力

- 计算机是一门更新速度很快的学科，每年都有新的技术不断涌现。因此，错位这个领域的从业人员的软件工程师需要具备很强的学习能力，否则时间一长就会跟不上技术进步的步伐。
- 很多 IT 公司在面试的时候会考察应聘者的学习能力。只有具备很强的学习能力及学习愿望的人，才能不断完善自己的知识结构，不断学习新的先进技术，让自己的职业生涯保持长久的生命力。
- 通常考察学习能力的方式。1. 询问应聘者最近在看什么书或者在做什么项目、从中学到了哪些技术。这些问题可以了解应聘者的学习愿望和学习能力。2. 能不能再较短的时间内理解一个新概念并解决相关的问题。如数组的旋转、二叉树镜像、丑数等。

善于学习、沟通的人也善于提问。

- 建议应聘者在面试过程中遇到不明白的地方多提问，这样面试官会觉得你态度积极、求职欲望强烈。

### 6.3 知识迁移能力

- 面试官考察知识迁移能力的方法：1. 把经典的问题稍作变换。期待应聘者能够找到和经典问题的联系，把解决经典问题的思路迁移过来解决新的问题。2. 先问一个简单的问题，再追问一个相关的同事难度更大的问题。希望应聘者能够总结前面解决简单问题的经验，把前面的思路、方法迁移过来。
- 知识迁移能力的通俗说法是“举一反三”的能力。**题目总是做不完的，我们不可能把所有面试题都准备一遍。因此，更重要的是没做一道面试题的时候，都要总结这道题的解法有什么特点，有哪些思路是可以应用到同类型的题目中去的。**

### 6.4 抽象建模能力

- 计算机只是一种工具，他被用来解决实际生产生活中的问题。程序员的工作就是把各种现实问题抽象成熟悉模型并用计算机的编程语言表达出来，因此有些面试官喜欢从日常生活中抽取提炼出问题考察应聘者是否能建立数学模型并解决问题。要解决这类型问题，应聘者除了需要具备杂扎实的数学基础和编程能力，还需要具有敏锐的洞察力和丰富的想象力。
- 建模的第一步是选择合理的数据结构来表述问题。我们根据问题的特点综合考虑性能、编程难度等因素之后，选择最合适的数据结构来表述问题，及建立模型。
- 建模的第二部是分析模型中的内在规律，并用编程语言表述这种规律。

### 6.5 发散思维能力

- 发散思维的特点是思维活动的多向性和变通性，也就是我们在思考问题时注重运用多思路、多方案、多途径来解决问题。对于同一问题，我们可以从不同的方向、侧面和层次，采用探索、转换、迁移、组合和分解等方法，提出多种创新的解法。
- 通过考查发散能力，可以了解应聘者探索新思路的激情；了解应聘者的灵活性和变通性；了解应聘则知识面的广度和深度。
